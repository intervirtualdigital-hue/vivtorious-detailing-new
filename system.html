What this website includes

This document is a behind-the-scenes summary of the Victorious Mobile Detailing site. It explains what's live in the V1 mockup, what's already wired up in code (waiting on a backend), what gets tracked, and what's planned next.

The V1 you're looking at is a static Vercel deploy — a polished visual mockup so you can see the finished feel of the site. The packages, prices, photos, and live admin tools are placeholders or dormant; we'll fill them in once the brand-side details are finalized.

Quiz funnel flow

The user journey was preserved from the source funnel and rebranded for Victorious:

  1. Homepage — user picks vehicle type (Coupe / Medium SUV / Large SUV-Truck / Lifted Truck). Saved to localStorage.victoriousQuiz.
  2. Packages — chooses a tier. (V1 is informational; pricing locks in later.)
  3. Pricing — shows their selection summary; redirects to text/DM CTA.
  4. Booking — user texts (646) 371-3739 or DMs @victoriousdetailing1.
  5. Thank-you — confirmation, next-step expectations, "back to home."

Admin dashboard dormant in V1

The original platform shipped with a full Express + SQLite admin. The code is preserved in server.js and views/admin/*.ejs so it can be turned on later by deploying to Railway or migrating SQLite to Postgres on Vercel. Today's static deploy doesn't run it.

RouteWhat it does
/admin/loginBcrypt-hashed admin password. Per-IP rate limit (5 attempts / 15 min).
/adminOverview cards: unique sessions, quiz starts, package selects, bookings — with 1d / 7d / 30d breakdowns and a 30-day visit sparkline.
/admin/funnel5-step funnel chart with drop-off rates: page_view → vehicle_select → package_select → booking_widget_loaded → booking_complete.
/admin/eventsRaw event log with type filter chips. Type counts. CSV export at /admin/events.csv.
/admin/pricingLive pricing matrix (4 packages × 4 vehicle types = 16 cells). Edit-in-place. Full audit history (old/new prices, who, when).
/admin/citiesSEO city-page CMS — toggle active, edit meta title/description, H1, intro copy, why-choose copy. Pages render at /detailing/:slug.

What gets tracked

Every page loads the first-party tracker at /js/tracking.js which auto-fires page_view. The vehicle-card click fires vehicle_select. Cookie-banner consent fires cookie_consent. The thank-you page fires booking_complete with vehicle/package/price props.

The endpoint on the backend (when on) is POST /api/event, which inserts into a SQLite events table with the following fields:

  • Event identity: type, page, ts, session_id (UUID stored in localStorage)
  • Quiz context: vehicle_type, package_name, price_total, city
  • Marketing attribution: utm_source, utm_medium, utm_campaign, utm_content, utm_term (auto-captured from URL, persisted across pages)
  • Session metadata: ip_hash (SHA-256 + secret salt — never raw IPs), user_agent, referrer, consent (accept/reject/unset)

Per-IP rate limit: 120 events/min. CSV export of last 50,000 events available from /admin/events.csv.

Tech stack

Front-end live

Plain HTML, CSS, and vanilla JS — no framework. Cinzel + Montserrat fonts. Mobile-first, responsive, lazy-loaded gallery images.

Hosting live

Vercel static deploy (Edge CDN). Custom rewrites for clean URLs in vercel.json. .vercelignore excludes the Express server code from the build.

Backend dormant

Express + EJS + better-sqlite3 + helmet + compression + sessions. Preserved in server.js for a later Railway / Postgres-on-Vercel deploy.

SEO live

LocalBusiness JSON-LD schema, OG and Twitter cards, canonical URLs, per-page meta. Sitemap and robots.txt available once the backend is live.

Privacy live

Cookie consent banner (accept / reject) writing to localStorage.victorious_consent. Privacy + Terms pages.

Pixel placeholder
